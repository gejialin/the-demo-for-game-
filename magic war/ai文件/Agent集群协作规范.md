# 《术士之战》Agent集群协作规范

## 1. 目的

本规范用于《术士之战》项目的固定角色 agent 并行协作。  
目标是让并行开发可执行、可验收、可追踪，避免多人同时改同一块内容造成冲突。

本项目默认采用 4 个固定角色：

1. 战斗Agent
2. 场景Agent
3. UIAgent
4. 验证记录Agent

主控 Agent 不属于以上 4 个执行角色中的任意一个，负责统一拆分、调度、验收、整合与正式记录。

---

## 2. 固定角色与职责

### 2.1 战斗Agent

负责内容：

- 角色控制
- 技能释放
- 战斗状态机
- AI 与目标选择
- 胜负流程
- 战斗相关配置接入

优先作用域：

- `Assets/script/charcter`
- `Assets/script/combat`
- `Assets/script/core` 中与战斗流程直接相关的文件

不负责：

- HUD 排版与选人界面
- 场地塌陷、岩浆、环境反馈主逻辑
- 正式日志落档

### 2.2 场景Agent

负责内容：

- 方形场地规则
- 岩浆区域与碰撞实体
- 塌陷式缩圈
- 安全区/危险区空间规则
- 环境反馈与场景表现配置

优先作用域：

- `Assets/script/Environment`
- 与场地规则直接相关的配置资产

不负责：

- UI 面板、HUD 文案
- 角色技能核心逻辑
- 正式日志落档

### 2.3 UIAgent

负责内容：

- HUD
- 选英雄界面
- 结果界面
- 中文文案接入
- 后续外文包预留结构

优先作用域：

- `Assets/script/UI`
- `Assets/script/core` 中仅与界面展示、文案映射、界面状态同步相关的文件

不负责：

- 场地环境主逻辑
- 角色战斗数值与技能逻辑
- 正式日志落档

### 2.4 验证记录Agent

负责内容：

- 编译验证
- 运行链路检查
- Unity MCP 验证辅助
- 截图路径、验证结果、待复核项整理
- 协作文档、模板、验证草稿整理

优先作用域：

- `ai文件`
- 验证脚本或验证说明文档

不负责：

- 直接改战斗主逻辑
- 直接改场景主逻辑
- 直接改 UI 主逻辑
- 直接向正式日志 `ai文件/操作记录.md` 追加内容

如确需越界改动，必须由主控 Agent 明确授权。

---

## 3. 文件所有权边界

### 3.1 默认边界

- 战斗Agent 不碰 `Assets/script/UI`
- 场景Agent 不碰 `Assets/script/UI`
- UIAgent 不碰 `Assets/script/Environment`
- 验证记录Agent 不直接改 `Assets/script/charcter`
- 验证记录Agent 不直接改 `Assets/script/combat`
- 验证记录Agent 不直接改 `Assets/script/Environment`

### 3.2 core 文件归属规则

`Assets/script/core` 不做长期固定归属，但同一轮任务中：

- 一个具体 `core` 文件只能分配给一个 agent 写入
- 其他 agent 可以阅读，但不能并行写同一文件
- 若 `core` 文件同时影响战斗与 UI，由主控 Agent 明确指定当前轮归属

### 3.3 正式日志归属规则

`ai文件/操作记录.md` 为主控 Agent 单点写入文件：

- 子 agent 不直接追加
- 子 agent 只提交草稿、验证结论、截图路径、待确认项
- 主控 Agent 汇总后按时间顺序写入正式日志

---

## 4. 什么时候适合并行

满足以下条件时，建议启用 2 到 4 个 agent 并行：

1. 任务可拆成至少 2 个互不重叠的子问题
2. 每个子问题都有清晰输出
3. 写入范围基本分离
4. 主控 Agent 可以在末尾统一验收

适合并行的典型情形：

- 新功能同时涉及战斗、场景、UI
- 一名 agent 做实现，另一名 agent 做验证
- 一名 agent 做中文 UI 接入，另一名 agent 做主逻辑接入
- 一名 agent 做配置资产整理，另一名 agent 做展示层接入
- 一名 agent 负责协作文档和验证模板，主线程继续做集成

---

## 5. 什么时候不适合并行

以下情况默认不启用集群并行，或先降级为单线程：

1. 多人会同时改同一个核心文件
2. 任务只是一个很小的单点修复
3. Unity MCP 当前连接不稳定，需要先排障
4. 正式日志整理阶段
5. 需求本身还没定清楚，拆分后容易反复返工

对于 `操作记录.md`、单个核心状态机文件、单个场景控制总入口文件，默认优先单线程处理。

---

## 6. 当前项目的默认协作流程

结合《术士之战》当前工程结构，默认流程如下：

1. 主控 Agent 先确认本轮目标与影响目录
2. 判断是否值得并行
3. 若启用并行，按固定角色分发任务
4. 每个 agent 只在授权目录内产出结果
5. 主控 Agent 在主线程继续做不重叠工作，或等待关键结果回收
6. 回收后检查是否有越界改动、冲突改动、遗漏验证
7. 主控 Agent 统一执行编译、运行态复核、日志入档

---

## 7. 主控 Agent 的统一验收职责

主控 Agent 负责最终验收，至少覆盖以下内容：

### 7.1 代码层

- `dotnet build Assembly-CSharp.csproj` 通过
- 无新增编译错误
- 无明显越界改动
- 同一轮未发生多人写同一核心文件

### 7.2 Unity 运行层

- 本轮关键主流程可以跑通
- MCP 可用时，优先补做场景或运行态验证
- 至少验证本轮改动直接影响到的链路

### 7.3 UI 层

- 新增可见文案默认中文
- 无明显英文残留，除非明确作为未来语言包占位
- 无明显文本重叠、布局挤压、状态不刷新

### 7.4 日志层

- 正式操作记录只由主控 Agent 写入 `ai文件/操作记录.md`
- 本轮各 agent 的结果已经被主控整理归档

---

## 8. 中文 UI 优先与外文包预留

本项目当前规则：

- 默认中文 UI 优先
- 可为未来外文包预留结构
- 当前阶段不强制建设完整 i18n 系统
- 新增用户可见文本时，优先提供中文显示内容

执行口径：

- UIAgent 负责检查新增界面是否有英文残留
- 战斗Agent 和 场景Agent 若新增用户可见文本，默认交付中文文案或可映射键值
- 若后续抽语言包，应优先从集中配置资产或统一文本入口接入，而不是在多个脚本里分散写死

---

## 9. 数据与配置原则

结合本项目已有要求，默认遵守以下规则：

1. 能进配置的数值尽量进配置
2. 场地尺寸、塌陷节奏、岩浆伤害、倒计时等，不应散落在多个脚本里重复写死
3. 文案与界面标签尽量通过集中入口管理
4. 若某轮任务只能先在 Unity Inspector 调整，也要保证后续可迁移到统一配置资产

---

## 10. 本规范的使用方式

本文件是项目内协作约定，不替代实际任务分配。  
每轮并行任务开始前，主控 Agent 应配合任务分发表一起使用，明确：

- 本轮启用了哪些角色
- 每个角色写哪些目录
- 哪些文件禁止碰
- 回收时要交什么结果

建议与 `ai文件/Agent任务分发表模板.md` 配套使用。
