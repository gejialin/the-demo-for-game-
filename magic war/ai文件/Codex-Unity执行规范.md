# 《术士之战》Codex-Unity执行规范

## 1. 目的

这份规范用于约束《术士之战》项目后续由 Codex 主控 Agent 与固定子 Agent 协作时的执行方式。

目标不是只让代码“能编译”，而是让每一轮改动都能做到：

1. 有明确分工
2. 有明确验证
3. 有明确记录
4. 有明确收口

---

## 2. 固定协作角色

本项目固定使用以下 4 类子 Agent：

1. 战斗Agent
   - 负责角色、技能、AI、胜负流程、战斗状态
   - 主要目录：
     - `Assets/script/charcter`
     - `Assets/script/combat`
     - 部分 `Assets/script/core`

2. 场景Agent
   - 负责场地、塌陷、岩浆、边界、空间规则、环境反馈
   - 主要目录：
     - `Assets/script/Environment`
     - 相关配置资产

3. UIAgent
   - 负责 HUD、选人界面、结果层、中文 UI、语言包预留
   - 主要目录：
     - `Assets/script/UI`
     - 部分 `Assets/script/core`

4. 验证记录Agent
   - 负责编译检查、运行验证清单、MCP 验证辅助、截图路径整理、日志草稿整理
   - 默认不直接修改主逻辑文件

主控 Agent 负责：

1. 拆任务
2. 分边界
3. 控冲突
4. 做集成
5. 做最终验收
6. 写正式操作记录

---

## 3. 并行启用规则

只有满足以下条件时，才启用多 Agent 并行：

1. 任务可拆成至少 2 个不重叠子问题
2. 每个子问题都有清晰输出
3. 各 Agent 写入范围基本分离
4. 主控 Agent 能在末尾统一验收

以下情况优先启用并行：

1. 同时涉及战斗、场地、UI
2. 一边实现，一边做只读验证
3. 一边接逻辑，一边整理配置或文档

以下情况优先单线程：

1. 同时要改同一个核心文件
2. 只是很小的单点修复
3. Unity MCP 当前连接不稳定
4. 只是在整理正式日志

---

## 4. 文件所有权规则

默认边界如下：

1. 战斗Agent 不碰 `Assets/script/UI`
2. 场景Agent 不碰 `Assets/script/UI`
3. UIAgent 不碰 `Assets/script/Environment`
4. 验证记录Agent 不直接改战斗/场景/UI核心逻辑
5. `Assets/script/core` 中同一个文件，同一轮只允许 1 个 Agent 写

正式日志文件：

- `D:\术士之战\magic war\ai文件\操作记录.md`

只允许主控 Agent 写入正式内容，其他 Agent 只能提交草稿或检查结果。

---

## 5. 执行顺序标准

每一轮任务默认按下面顺序推进：

1. 主控确认目标
2. 判断是否需要启用并行
3. 明确本轮文件边界
4. 子 Agent 并行执行或只读审查
5. 主控完成集成
6. 跑代码层验证
7. 有 Unity 会话时跑运行层验证
8. 更新 `操作记录.md`
9. 汇报本轮结果与未闭环项

---

## 6. 数据与配置原则

本项目后续遵守：

1. 能放配置的数值，不优先写死在逻辑里
2. 场地尺寸、塌陷节奏、岩浆伤害、倒计时等优先进配置
3. 用户可见文本优先走集中入口
4. 当前默认中文 UI，外文包只做结构预留
5. 若某轮先在 Unity Inspector 调值，后续应尽量回收到统一配置资产

---

## 7. 验收分层标准

每一轮至少明确区分下面两类结论：

### 7.1 代码层通过

代码层通过只代表：

1. `dotnet build Assembly-CSharp.csproj` 通过
2. 没有新增编译错误
3. 逻辑入口能连上

代码层通过不代表：

1. 碰撞一定正确
2. 高度一定正确
3. 重开流程一定闭环
4. HUD 一定同步
5. 手感一定符合预期

### 7.2 运行层通过

运行层通过必须依赖 Unity 在线验证，至少确认：

1. 实际场景表现正确
2. 碰撞/高度/状态同步正确
3. 核心链路可以完整跑通

没有做 Unity 在线验证时，结论必须明确写成：

- “代码层已验证，运行层未验证”

---

## 8. 验证记录Agent标准职责

验证记录Agent 每轮默认检查以下内容：

1. 本轮改动影响了哪些链路
2. 哪些结论可以靠编译验证
3. 哪些结论必须进 Unity 验证
4. 需要补哪些截图
5. 哪些问题阻塞继续开发
6. 哪些问题可以后续细化

验证记录Agent 默认输出：

1. 本轮最小验收清单
2. 待 Unity 在线时验证项
3. 截图建议
4. 操作记录草稿要点

---

## 9. 内部测试最小清单

后续每轮至少按影响范围检查以下项目：

### 9.1 场地/塌陷/岩浆

1. 场地与危险区碰撞是否连贯
2. 塌陷后是否真的由岩浆替代
3. 人物是否会掉进空洞
4. 岩浆高度是否符合设计预期
5. 边缘是否存在缝隙

### 9.2 战局流程

1. 倒计时是否正确开始
2. 战斗是否正常进入 Active
3. 一局结束是否正常弹结果层
4. 重新开始是否重置地图
5. 重新开始是否重置计时器

### 9.3 角色与技能

1. 当前角色是否正确生成
2. 敌方是否按规则生成
3. 主动技能是否能释放
4. 近战/远程/受击/死亡链路是否闭环

### 9.4 UI

1. 新增可见文本默认中文
2. 无明显英文残留
3. 无文本重叠
4. HUD 状态是否同步刷新

### 9.5 日志

1. 本轮改了什么
2. 本轮验证到什么层级
3. 还有什么待 Unity 在线验证

---

## 10. 场地验收标准

凡是改到 `Assets/script/Environment`，至少补查：

1. 安全区尺寸是否正确
2. 塌陷触发时机是否正确
3. 塌陷后碰撞是否接续
4. 岩浆伤害与灼烧是否正常
5. 重开后场地是否完全恢复

---

## 11. 重开流程验收标准

凡是改到战局重开、返回选人、结果层，至少补查：

1. 敌人是否清理
2. 玩家是否重生
3. 地图是否恢复
4. 塌陷计时是否重置
5. 战斗计时是否重置
6. HUD 文案是否恢复到初始状态
7. 不应残留上一局状态

---

## 12. 文档与日志标准

后续统一使用：

1. 正式日志：
   - `D:\术士之战\magic war\ai文件\操作记录.md`

2. 协作文档：
   - `D:\术士之战\magic war\ai文件\Agent集群协作规范.md`
   - `D:\术士之战\magic war\ai文件\Agent任务分发表模板.md`
   - `D:\术士之战\magic war\ai文件\待Unity在线时验证清单-2026-06-20.md`
   - `D:\术士之战\magic war\ai文件\Codex-Unity执行规范.md`
   - `D:\术士之战\magic war\ai文件\内部测试Agent执行清单.md`

日志写法要求：

1. 按当天实际工作顺序写
2. 不删除旧记录
3. 明确区分“已验证”和“待验证”
4. 不把子 Agent 草稿直接原样贴进正式日志

---

## 13. 后续固定动作

从这份规范生效后，每一轮默认执行：

1. 主控先定边界
2. 能并行就启用固定 Agent
3. 至少有一个验证记录Agent跟进
4. 每轮结束更新正式日志
5. 若 Unity 不在线，明确留下待验证清单

这套规则的重点不是追求形式，而是减少：

1. 编译过了但运行翻车
2. 修了一个点又破另一个点
3. 多 Agent 写乱
4. 日志与真实进展脱节
