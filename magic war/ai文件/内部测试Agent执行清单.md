# 《术士之战》内部测试Agent执行清单

## 1. 文档用途

这份文档给固定的“验证记录Agent / 内部测试Agent”使用。

它的职责不是代替主控做开发，而是把每轮改动后的验证工作做成固定动作，减少下面这些问题：

1. 编译通过，但运行翻车
2. 场地、计时器、UI 状态不同步
3. 改了一个点，带坏另一条主链路
4. 最后没有证据，日志只能靠回忆补

---

## 2. 内部测试Agent职责

每轮改动后，内部测试Agent默认负责：

1. 检查本轮影响链路
2. 区分哪些能靠编译验证，哪些必须进 Unity 验证
3. 输出本轮最小验收项
4. 记录截图建议和验证证据
5. 整理待复核项和阻塞项
6. 向主控提交验证草稿

内部测试Agent默认不负责：

1. 直接改主逻辑代码
2. 直接写正式 `操作记录.md`
3. 代替主控做最终结论

---

## 3. 每轮固定检查项

每轮改动后，至少检查下面 10 项：

1. `Assembly-CSharp` 是否编译通过
2. 本轮影响到的 scene / prefab / config / Resources 文件是否仍存在
3. 开局链路是否能走通
4. 本轮改动直连链路是否复测
5. Unity Console 是否出现新增 Error
6. 新增 Warning 是否已归因
7. 运行时生成对象是否可重复进入而不堆积
8. 时间状态是否正确恢复
9. 中文 UI 是否正常显示
10. 验证证据是否齐全

---

## 4. 编译验证与运行验证边界

### 4.1 可先靠编译验证的问题

1. C# 语法错误
2. 类型错误
3. 方法签名改坏
4. 字段名、类名、命名空间引用错误
5. 明显的代码依赖断裂

### 4.2 必须进 Unity 验证的问题

1. 场地生成是否正确
2. 塌陷时机是否正确
3. 岩浆碰撞与伤害是否正确
4. 重新开始是否重置完整
5. 返回选人是否收干净
6. HUD 是否刷新正常
7. 中文 UI 是否有乱码、重叠、截断
8. prefab / scene / asset 引用是否挂对
9. Resources 配置是否真的加载生效
10. 玩家实际手感与空间规则是否符合预期

结论规则：

1. 没进 Unity 跑过的，只能写“代码层通过”
2. 进 Unity 跑过且主链路通过的，才能写“运行层通过”

---

## 5. 最小验收矩阵

### 5.1 场地

最小验收项：

1. 黑色安全区可见
2. 岩浆环带可见
3. 玩家与敌人出生在安全区
4. 不出现旧地面重复生成

### 5.2 塌陷

最小验收项：

1. 至少发生一次真实塌陷
2. HUD 有塌陷/倒计时提示
3. 塌陷后安全区边界真实变化
4. 塌陷区不再被当作安全地面

### 5.3 岩浆

最小验收项：

1. 接触岩浆后按 tick 掉血
2. 可触发燃烧
3. HUD 状态可显示 Lava / Burn
4. 离开岩浆后接触伤害停止
5. 燃烧按持续时间结束

### 5.4 重新开始

最小验收项：

1. 结算层出现“重新开始”
2. 点击后旧敌人清除
3. 当前职业重新生成
4. 地图恢复
5. 塌陷计时重置
6. 战斗计时重置
7. 结果层关闭

### 5.5 返回选人

最小验收项：

1. 点击后回到选人界面
2. 不残留上一局玩家和敌人
3. 不残留暂停状态
4. 不残留结果层

### 5.6 中文 UI

最小验收项：

1. 选人、暂停、结算、HUD 关键文案默认中文
2. 无明显英文残留
3. 无乱码
4. 无明显文本重叠和截断

---

## 6. 特殊改动的强制复测规则

### 6.1 改到 `Environment`

至少补测：

1. 场地生成
2. 塌陷
3. 岩浆接触
4. 重开后场地恢复

### 6.2 改到 `BattleFlowController` 或相关战局流程

至少补测：

1. 开局倒计时
2. 战斗进入 Active
3. 结算
4. 重新开始
5. 返回选人

### 6.3 改到 `UI` 或中文文案

至少补测：

1. 关键中文文本是否刷新
2. 是否有英文残留
3. 是否有乱码 / 重叠 / 挤压

### 6.4 改到技能、状态、受击或 Lava/Burn

至少补测：

1. 释放链路
2. 掉血链路
3. HUD 状态同步
4. 死亡/重开后状态清理

---

## 7. 失败判定

下面任一出现，默认判为本轮未通过：

1. 编译失败
2. Unity Console 出现本轮新增 Error
3. 主链路无法从开局跑到结算
4. 重新开始不能恢复初始状态
5. 场地/塌陷/岩浆碰撞出现明显错误
6. 用户可见中文 UI 出现严重乱码或重叠
7. 运行时对象重复堆积

下面情况可判“部分通过”：

1. 代码层通过
2. 主链路能跑
3. 存在非阻塞 Warning 或轻微表现问题
4. 已明确待 Unity 进一步复核项

---

## 8. 验证记录模板

每轮建议至少输出下面这些内容给主控：

1. 验证项：
2. 结果：通过 / 不通过 / 部分通过
3. 现象：
4. 证据：
5. 截图路径：
6. 是否阻塞：
7. 待复核项：

---

## 9. 与正式日志的关系

这份清单是内部验证文档，不是正式操作日志。

规则如下：

1. 内部测试Agent 只提交验证草稿
2. 主控 Agent 负责写入：
   - `D:\术士之战\magic war\ai文件\操作记录.md`
3. 正式日志必须按当天实际工作顺序整理
4. 未做 Unity 运行验证的项，不允许写成“功能已完成”

---

## 10. 默认执行口径

从这份文档生效后：

1. 只要任务值得并行，默认单开一个内部测试Agent
2. 内部测试Agent优先做只读审查和验收草稿
3. 每轮结束前，主控至少要回答：
   - 代码层验证到了吗
   - 运行层验证到了吗
   - 还有什么待 Unity 在线验证

这套清单的目标只有一个：

让“基础纰漏”尽量在内部测试阶段暴露，而不是总靠你打完一把才发现。
